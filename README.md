# local-code-agent

*Read this in [English](README.en.md).*

Локальний кодинг-агент на **gemma4:26b-a4b-it-qat** (через Ollama), спроєктований
під слабку локальну модель: вузький контекст, декомпозиція задачі, агентний
tool-loop і детерміноване поводження з великими даними.

## Ідея

- **Planner** (think=on) — читає задачу й ріже її на дрібні кроки.
- **Per-step tool-loop** (think=off, вузький контекст) — на кожен крок модель
  через нативний tool-calling сама викликає інструменти (читати/писати/редагувати
  файли, запускати команди), а рантайм їх виконує й повертає результат, поки крок
  не завершено. Модель ніколи не виконує код сама — це робить наш цикл.
- **Strip/Restore** — великі опакові літерали (напр. протобафи в коді) детерміновано
  вирізаються в плейсхолдери *до* LLM і повертаються байт-у-байт *після*. Модель їх
  не бачить → не псує, а контекст не роздувається.

## Чому не Aider

Aider завжди кладе весь файл у контекст і регенерує його цілком — на токен-щільних
файлах слабка локальна модель захлинається й псує дані. Тут великі дані взагалі не
йдуть крізь модель.

## Реєстр інструментів

Усе виконання уніфіковане під tool-calling. Інструмент = `{схема, обробник}`;
додати новий = зареєструвати його в реєстрі, без змін у циклі. Базові інструменти:

| Інструмент | Призначення |
|---|---|
| `list_dir` | структура файлів проєкту |
| `read_file` | прочитати файл |
| `write_file` | створити / перезаписати файл |
| `edit_file` | відрефакторити наявний `.py` (зі strip/restore) |
| `create_from_source` | створити НОВИЙ файл як копію джерела (зі strip/restore) |
| `run_shell` | запустити програму (python / pytest / git) |

## Дозволи

- **правки** (`ask` / `auto`) — `ask` показує план і чекає підтвердження;
  `auto` виконує одразу.
- **консоль** (`allowlist` / `ask` / `off`) — `allowlist` дозволяє лише безпечні
  команди; `ask` питає попапом перед кожною командою; `off` вимикає shell.

## Структура

```
agent/
  config.py      профілі EXECUTOR / PLANNER
  llm.py         клієнт Ollama (tool-calling, think-перемикач)
  literals.py    strip / restore великих літералів
  tools.py       детерміновані файлові/shell примітиви + allow-list
  executor.py    run_edit / create_from_source (strip→LLM→restore→diff)
  toolkit.py     реєстр інструментів + ToolContext (root, дозволи, confirm)
  agent_loop.py  per-step tool-loop (run_step)
  planner.py     декомпозиція задачі на кроки
  intent.py      класифікація наміру (edit / plan / answer / shell)
  answerer.py    відповіді/аналіз коду (think=on)
  memory.py      стан задачі на диск
  session.py     сесії/чати (проєкт, дозволи, історія, джерела)
  project.py     AGENT.md (знання проєкту) + скан структури
lca_web/
  lca_web.py     веб-інтерфейс (Reflex, стиль Claude)
main.py          точка входу (зручно з PyCharm)
rxconfig.py      конфіг Reflex
```

## Знання проєкту (AGENT.md)

У корені робочого проєкту може лежати `AGENT.md` — куроване знання, що інжектиться
планувальнику/виконавцю на кожну задачу (не RAG/ембединги — для малого проєкту й
слабкої моделі один курований файл працює краще). Опційний init-scan авто-чернетує
`AGENT.md` для нового репозиторію.

## Вимоги

- Python 3.12, `requests`, `reflex==0.9.5.post2`
- Ollama з моделлю `gemma4:26b-a4b-it-qat`

## Запуск

```powershell
# веб-інтерфейс (фронтенд :3000, бекенд :8000)
.\run_web.ps1
# або
python main.py
```

З PyCharm: відкрий `main.py` → ▶ Run (інтерпретатор — `.venv` проєкту).
